The correction this surface serves

Review first, export second

Flow Editor→Export PDF✗ wrong shape

Flow Editor→Print Preview→Review real pages→Export / Print✓

Print Preview is the gate before export. The director/writer sees the screenplay as a package — real pages, real geometry — and only then ships it. Export and Print become actions inside this surface, not menu verbs that bypass review.

Observation — the audit's finding

The pages are already real: the pipeline (Normalizer → ManuscriptGeometry → PageMap → RenderModel → PrintRenderer) paints one inch-true .rga-page-sheet per page, mirrors for RTL, and the PDF pipe works underneath. What's missing is the room around the pages — today Print Preview is a bare stack of sheets on a dark overlay with zero visible chrome, reachable only by Esc and undiscoverable PageUp/PageDown.

01 · Review-Surface Identity

A room you step into, not a panel you toggle

Print Preview is the moment the draft stops being editable text and becomes a deliverable — the room you enter to review the package before it leaves your hands. The closest analogue is the lock review at a production office: the script held up, read as pages, checked before it's sent.

  • A deliberate shift of mode. Entering recedes the writing chrome and gives the pages the stage. Reverent, full-attention review — not a glance.
  • Read-only by nature. Nothing here edits the script — no dirty state, no save anxiety, no inline editing. You look, navigate, and ship or step back.
  • A ViewManager peer mode, dressed as a first-class surface. Keep the lifecycle simple (mutual exclusion, restore prior view on exit); own bespoke chrome rather than borrowing the shell's. The "peer vs. workspace tab" question stays closed in this arc.
Three things it must never be

Not a browser print dialog (no Destination/Pages/Layout sidebar). Not a generic OS print panel. Not a dashboard (no stat tiles, no metadata panels). It is a screenplay page-review surface — the page is the hero and the chrome whispers.

02 · Top Toolbar Anatomy

One slim review bar, three zones

The central missing piece is a persistent chrome host that is not one of the scrolling sheets. Direction: one slim top bar, floating above the sheet stack, always present — controls up and out of the way so the page stays uninterrupted.

03 · Page Navigation

The indicator is the spine

All of this is backed by data that already exists — RenderModel.totalPages, ScriptSession.currentPage, and the current-sheet index _scrollByOneSheet already computes but never reports.

  • Live current-page indicator. A monospaced 3 / 12 at the bar's center, updating as the writer scrolls. The "you are on page N of the package" signal — the navigation cluster's gravitational center.
  • Prev / next. Quiet chevrons flanking the indicator, each scrolling one sheet via the existing mechanism. PageUp/PageDown keep working and gain a discoverable hint.
  • Page count. The / 12 from totalPages — one honest total.
  • Jump = the indicator itself. Click the number, it becomes a small mono input; type 7, Enter → page 7. No separate go-to modal; the indicator and the jump are one object. Esc cancels.
  • Keyboard: PageUp/PageDown, Home/End, and the jump field — surfaced as a quiet, discoverable hint.

04 · Zoom / Fit

Fit page on entry; transform only

  • Fit page — the default. A director wants to see the whole page as a page — composition, white space, where scenes break. Entering lands here.
  • Fit width — the reading mode. Fills the horizontal space to read action and dialogue, scrolling vertically through the package.
  • Zoom in/out — secondary. A − 100% + stepper with a readout for inspecting detail. A stepper + fit buttons, never a continuous slider (dashboard-y for this surface).
Direction — the hard constraint

Zoom and fit are presentation transforms only (CSS scale on the sheet stack). They must never feed back into PageMap or inch-truth. The page geometry is the single source of truth; zoom changes how big the page looks, never what the page is.

05 · Export / Print Actions

The commit edge — review's whole point

Export is downstream of review. These live in the output zone (inline-end) and are the only "loud" elements in an otherwise quiet surface.

  • Export PDF — primary, weighted. A filled accent button — the one place a confident fill belongs. Routes to the working Rga.PdfExport.run() pipe; no backend change.
  • A light review-aware popover — not a modal, not silent. Silent one-tap export undercuts the "review first" correction; a heavy generic modal becomes the print dialog the brief forbids. Between them: a compact options popover anchored to the button — filename, page range, and a place that grows as package features land (title page, scene-number draft, revision color — deferred). Designed to expand gracefully, with an explicit "Export" confirm.
  • Print — present the slot, defer the wire. Native print / @media print is a deferred slice. Print sits beside Export as a sibling action but is shown-but-deferred until the print path lands. Design the home; don't assume the pipe.

06 · Exit / Back-to-Editor

A visible step back out of the room

  • A visible exit in the context zone. A ← Done / ← Editor affordance — label + arrow reads clearer than a bare ✕ for "step back to writing." Keep Esc as the keyboard path.
  • Returns to the prior view exactly. ViewManager restores Flow/Draft and the writer's cursor and scroll — stepping out lands you precisely where you left writing. Already implemented; preserve it.
  • Instant and safe. Read-only surface → no confirmation, no "discard?" prompt. A frictionless step back.

Direction — the one test

Every chrome decision is judged against one question: does the page still feel like the hero? If a control or panel makes the chrome compete with the paper, it is wrong for this surface.

08 · Page Thumbnails / Strip

Yes — collapsible left rail, deferred past v1

  • Where: a narrow vertical rail on the inline-start edge — mini-sheets with page numbers, click to jump, current page highlighted. The only side-chrome the surface has.
  • Off by default. The page is the hero; the strip is a power tool for long packages (100+ pages). Toggle from the top bar.
  • Deferred by necessity. Needs render-to-thumbnail + virtualization. Designed now, built later — the jump-to-page field covers navigation in v1.

09 · RTL / LTR Chrome

The chrome mirrors with the sheets

  • The whole top bar flows by logical direction. In RTL (Kurdish/Sorani, Arabic), zones flip: exit/context to the right, output/commit to the left — each on the natural forward edge.
  • Directional glyphs mirror; the page indicator and jump field stay logical (3 / 12 reads consistently).
  • Matches the editor's existing RTL discipline — logical properties throughout. Native in each direction, never a mirrored-content bolt-on.

10 · Empty / Error / Loading

Composure, because the page is the artifact

  • Loading: a calm, centered "Composing pages…" with page-size context — no heavy spinner, no progress-bar theatre. The pages are being laid out, briefly.
  • Empty: "Nothing to preview yet — write a scene to see pages." Centered, muted, no illustration, no CTA hero.
  • Error: "Couldn't compose the page preview," with Retry and Back-to-editor. Never a stack trace, never alarm-red. The writer's work is safe in Flow — the message should make that obvious.

11 · 12 · Title Page · MORE / CONT'D · Revision Marks

The package, anticipated

Title page (§11)

The title page is the cover of the package — currently an in-flow body block, not a cover sheet (deferred). When it lands, it becomes the first sheet in the reviewed stack (title, author, contact, draft date), page-number suppressed. The package reads cover-first; the indicator reads "Title," body numbering begins at 1. The surface's package framing absorbs it without rework.

MORE / CONT'D & revision marks (§12)

  • MORE / CONT'D — render in the sheet, no chrome. Continuation marks are page-output content; they appear within the sheets when the slice lands. The surface needs no control — it is simply the only place they can be verified.
  • Scene-number gutters & revision marks — toggleable review layers. Draft-state overlays the director may want on (production draft) or off (clean copy). A small "review layers" toggle group in the top bar (Scene numbers · Revision marks) shows/hides them on the sheets. Reserve the slot now; both are deferred slices.

Closing

01

Strongest existing foundation

The pure, well-tested rendering pipeline — the pages are already real and correct. Normalizer → ManuscriptGeometry.resolve → PageMap → RenderModel → PrintRenderer paints one inch-true .rga-page-sheet per page, never measuring DOM, with correct Letter/A4/Legal geometry and RTL binding-side mirroring — and Rga.PdfExport.run() works end-to-end underneath. Page-awareness data (totalPages, nav-index pages, ScriptSession.currentPage) already exists, merely unsurfaced. The redesign never touches page truth; it builds the room around pages that are already trustworthy.

02

Biggest missing UX piece

There is no chrome at all — and therefore no review workflow. While active, every shell control is hidden, leaving a bare stack of sheets with only Esc and undiscoverable PageUp/PageDown — no page count, no current-page tracking, no navigation affordance, no export from within. The single biggest piece is the persistent top review bar hosting page awareness, navigation, and Export — the element that converts a silent sheet stack into a reviewable, shippable package.

03

Recommended first implementation bundle

"Review Bar v1." The persistent top bar carrying: a visible exit (← Done); a live page indicator (N / total) with prev/next and jump-to-page; Fit page (default) / Fit width / zoom stepper; and Export PDF wired to the existing pipe with a light options popover. No thumbnails, no Print wire, no package features. Every element draws on data and pipes that already exist (audit §3), so the bundle is low-risk and high-payoff — it delivers the entire "review then export" correction in one coherent surface without touching page truth or any deferred slice.

04

Highest-risk design mistake to avoid

Turning the review surface into a browser print dialog, a generic export modal, or a dashboard — burying the page under system-style option panels, destination pickers, or stat tiles. The page must stay the hero and the chrome must whisper; the moment options out-shout the paper, the surface has failed. Two technical corollaries of the same mistake: letting zoom or chrome feed back into page truth (zoom is presentation-only; PageMap is the single resolver), and leaking page seams back into Flow (page truth lives here, never upstream in continuous drafting). Guard the page's primacy and the single-source-of-truth boundary above all.
