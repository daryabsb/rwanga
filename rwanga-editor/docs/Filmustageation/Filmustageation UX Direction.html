Section 1

Overall Workspace Philosophy

The Central Question

Every decision in this document answers one question: how should this editor feel while writing? Not "how should it look." Not "how many features can we surface." The feeling during a writing session is the product. Everything else is infrastructure.

Emotional Hierarchy

The workspace has four emotional layers. They are listed here in order of visual weight — heaviest first, lightest last. This is not the order of screen real estate. The heaviest layer occupies the most attention but may occupy relatively little space. The lightest may be large but should never demand focus.

Layer Feels like Behavior
The Page Sacred, stable, physically important Always present. Never shifts, slides, or resizes during writing. The gravitational center. Every other layer orbits this.
Contextual Intelligence A quiet observer in the room Appears only when relevant. Whispers, never shouts. Surfaces information the writer didn't know they needed — then retreats.
Session Awareness Ambient knowledge of where you are Scene count, position in the script, session continuity markers. Peripheral — you glance, you don't stare.
Shell Chrome Architecture — walls, not furniture The rail, toolbar, status bar. These are the room's walls. You don't notice good walls. You notice bad ones.

The "Living Script" Feeling

The doctrine defines the script as a session with its own brain. Notes, scenes, tags, revisions, AI conversations all travel with the script. The UX consequence: when you open a script, you are not opening a file. You are resuming a creative session. The editor should communicate this through continuity — your cursor position, your open panels, your last note, your most recent AI conversation — all restored silently. No "welcome back" splash screen. No loading sequence. Just: you're here again, where you left off.

Doctrine Anchor

"The script is no longer a flat file. It is a session with its own brain." — §7.1. Notes, scenes, tags, revisions, AI conversations all travel with the script. The editor's job is to make this feel inevitable, not technical.

Cinematic Atmosphere Without Costume

"Cinematic" does not mean dark gradients, lens flares, or film-grain textures. It means the composition principles of cinema applied to interface design: clear focal points, deliberate negative space, controlled depth of field (some things sharp, others ambient), and rhythm — scenes flow, panels breathe, transitions have purpose.

The workspace should feel like a production studio at 2 AM — focused, quiet, purposeful. Not a movie theater. Not a VFX suite. A place where serious creative work happens under controlled conditions.

What Should Feel Heavy

  • The page itself. The screenplay text is the heaviest object in the room. Its typography, its spacing, its presence — all communicate authority.
  • Scene boundaries. When you cross from one scene to another, the editor should acknowledge it. Not dramatically — structurally. Scene transitions are architectural events in a screenplay.

What Should Feel Invisible

  • Navigation between tools. Switching from writing to breakdown to notes should feel like turning your head, not changing applications.
  • Persistence. Autosave, session recovery, state preservation — all of this should be so reliable that the writer forgets it exists.
  • Plugin boundaries. The writer should never think "I am now in the screenplay plugin." They are writing. The system knows the rules.

What Should Feel Ambient

  • Production context. Character counts, prop lists, location data — all of this exists in the periphery. It's knowledge you accumulate by glancing, not by switching modes.
  • AI readiness. The intelligence layer should feel like a colleague sitting nearby — available if you turn to them, silent otherwise.
  • Session history. Your revision trail, your note timeline, your AI conversations — they form a sediment layer beneath the current writing surface. Present but not foregrounded.

Section 2

Shell Direction

Current State Observation

Observation — Screenshot Analysis

The current shell has three layers of chrome competing for the same role. The Filmustage top navigation bar (Script / Scheduling / Call Sheets / Budgeting / Storyboards) is platform navigation. Below it, a secondary toolbar (Scene Breaks, Download Synopsis, Share, AI Dude, Export) mixes document actions with platform features. The left sidebar is a breakdown inventory panel posing as a scene navigator. All three are "always on" — none adapts to the writer's current focus.

The shell must be decomposed into zones with clear ownership. The doctrine's three-layer architecture (Engine → Plugin → Platform) must be physically visible in the shell's structure — not as labels, but as behavior. Engine chrome stays permanent. Plugin chrome adapts to the active doc-type. Platform chrome lives outside the editor viewport entirely.

Shell Zone Map

The following diagram defines the intended spatial relationships and ownership of each shell zone:

Workspace Zone Ownership

Rail CORE
Sidebar CORE frame
PLUGIN contents

Zone-by-Zone Direction

Activity Rail — Stays minimal. 48px. Icon-only. This is the editor's spine — it provides access to sidebar views (scenes, tags/breakdown, search, extensions, settings) but does not itself contain content. The rail is always CORE. Its icon set changes subtly per doc-type — screenplay shows a clapboard scene icon; a future novel plugin would show chapter markers. But the rail structure never changes. The active-indicator (currently a left-edge accent bar) should use the section color system: Script orange when in writing mode, Breakdown blue when in breakdown view, etc.

Sidebar — This is where the biggest evolution must happen. Currently the sidebar is a static inventory accordion (CAST (5), PROPS (8), SET DRESSING (3)...). This is production data, not writing navigation. The sidebar's primary identity must be structural navigation — scenes, sequences, acts. The breakdown inventory is a secondary view, accessible via a rail icon switch, not the default.

Direction

The sidebar's default view when writing should be the scene navigator — a clean list of scenes with minimal metadata (number, slug, estimated duration). The breakdown categories accordion becomes a separate sidebar view activated from the rail, used during breakdown work, not during writing. This single change eliminates the "admin panel" feeling.

Inspector (Right Panel) — Currently shows "Notes: Scene 4" — a plain text box. The inspector should become the contextual intelligence surface. Its contents depend on what's selected or where the cursor is. Click a character name → inspector shows character details, occurrences, notes. Place cursor in a scene → inspector shows scene-level metadata, notes, AI observations. Select nothing → inspector shows script-level overview. The inspector is plugin-contributed content inside a CORE-owned frame. Default state: collapsed. The writing surface expands to fill the space. The inspector opens when the writer explicitly requests context.

Top Chrome — Must be radically simplified. The current toolbar (Scene Breaks, Download Synopsis, Share Project, Script AI Dude, Export) mixes document verbs, platform actions, and AI access into one bar. Direction: the title bar shows only the script name and session state. Document verbs (export, print) live in the menu system or command palette. Platform actions (share, sync) live at the platform layer — not inside the editor's toolbar. The "Script AI Dude" button should not exist as a persistent toolbar element. AI is accessed contextually, not via a chrome button.

Command Palette — This becomes the editor's primary power-user surface. All commands, document verbs, navigation, plugin actions — accessible via a single keyboard shortcut. The palette is CORE infrastructure; plugins register their commands into it. This replaces the need for toolbar buttons. The palette's visual grammar should follow the existing Rwanga design system: dark surface, amber focus indicators, monospace shortcut labels.

Status Bar — Remains a thin, ambient strip. CORE provides: page count, word count, cursor position, sync status. Plugins contribute additional slots: screenplay adds scene count and estimated screen time; a future research plugin adds citation count. The status bar uses the existing Rwanga design system's status-bar pattern — tinted background, compact type, information density without clutter. The status bar is where the section color system is most visible — its tint reflects the current workspace mode.

What Stays Minimal

  • The activity rail — icon-only, narrow, permanent
  • The status bar — one line, ambient, glanceable
  • The title bar — script name and session indicator only

What Becomes Adaptive

  • The sidebar — its content changes based on the rail selection and the active doc-type
  • The inspector — its panels change based on cursor position and selection

What Should Disappear During Writing Focus

  • The sidebar — collapsible to zero width; writing surface expands
  • The inspector — default collapsed; opened on demand
  • Any toolbar or action bar — the page rules alone during deep writing

Section 3

Plugin-Aware UX Direction

The Core Problem

Today, the editor is the screenplay plugin. Every surface, every icon, every panel assumes screenplay. The doctrine names this explicitly: "Screenplay is a doc-type, not the kernel." The UX must separate what belongs to the engine from what belongs to the plugin — without making the separation visible to the writer.

The paradox: the writer should never think about plugins. They open a screenplay and the editor knows screenplay rules. They open a novel and the editor knows novel rules. The mechanism is invisible. The experience is seamless. But the architecture beneath must be clean.

How Screenplay Mode Should Feel

When a screenplay document is open, the editor should feel like it was born for screenwriting. The formatting toolbar understands slug lines, action, dialogue, parenthetical, transition. The scene navigator in the sidebar speaks screenplay language. The inspector surfaces character and location data. The status bar shows page count in industry-standard units. Nothing feels "plugged in." It feels native.

How Future Plugins Coexist

A future novel plugin will replace the sidebar's scene navigator with a chapter navigator. The formatting toolbar will offer paragraph, heading, blockquote — not slug line and dialogue. The inspector will surface character arcs and timeline data instead of props and locations. The status bar will show word count and chapter position instead of page count and scene number.

The shell remains identical. Rail, sidebar frame, writing surface, inspector frame, status bar — same geometry, same behavior. What changes is the contents of each slot. This is the plugin contract made visible.

Plugin Slot Architecture — Same Shell, Different Contents
Shell (CORE)
Rail · Sidebar frame · Writing surface · Inspector frame · Status bar · Command palette · Title bar
Permanent. Domain-neutral. Never changes per doc-type.
Screenplay
Scene navigator → sidebar · Scene heading / action / dialogue → schema · Character/prop/location panels → inspector · Page count → status bar · FDX/PDF → export · Breakdown categories → sidebar (alt view)
Novel
Chapter navigator → sidebar · Paragraph / heading / blockquote → schema · Character arc / timeline → inspector · Word count → status bar · EPUB/DOCX → export · Outline view → sidebar (alt view)
Research
Section navigator → sidebar · Paragraph / citation / figure → schema · References / footnotes → inspector · Citation count → status bar · BibTeX/PDF → export · Outline view → sidebar (alt view)

Plugin Identity in the UI

Each plugin should have a subtle identity presence — not branding, but a recognizable feel. The section color system already provides this vocabulary: Script orange for screenplay, Breakdown blue for production work. A novel plugin might adopt a warm neutral; research a cool slate. This color appears in:

  • The active rail indicator
  • The status bar tint
  • The sidebar's section header
  • The command palette's plugin-prefixed commands

Never in the writing surface itself. The page is neutral ground.

Plugin Switching Conceptual Direction

Plugin switching is a document-level event, not a workspace-level mode switch. You don't "switch to novel mode." You open a novel document and the editor adapts. The New Document flow should offer doc-type selection as a calm choice — not a dramatic mode picker. Think: a clean list with doc-type icons and one-line descriptions. No splash screens, no onboarding wizards, no "Choose your experience" fanfare.

Doctrine Anchor

"Each plugin's purpose is to teach the engine a writing form. The engine learns; it does not get rewritten." — §6. The editor absorbs new writing forms without losing its identity. It becomes more capable with each plugin, never more confused.

Section 4

Screenplay Plugin UX Direction

What Works

  • The writing surface itself. Courier typography, proper screenplay block formatting (action full-width, character centered, dialogue indented, parentheticals narrower). This is the correct foundation. The page feels like a screenplay page.
  • Color-coded breakdown tags inline. The highlighted words (characters in blue, props in amber/orange, locations in green, etc.) are valuable — they surface production information within the writing context, not in a separate panel. This is the right instinct: intelligence embedded in the page, not extracted from it.
  • Scene numbering in the gutter area. Scene numbers as structural markers, positioned near the scene header — this is correct screenplay convention and helps with navigation.
  • The scene list in the left sidebar. The list of scenes (0: PRE-SCENE, 1: INT. EMPTY STUDIO, 2: EXT. DEVASTATED CITY, etc.) is useful navigation. Scene highlighting with color-coded bars for the selected scene works.

What Feels Too SaaS

Observation

The breakdown categories accordion (CAST (5), EXTRAS (1), PROPS (8), SET DRESSING (3), LOCATION DETAILS (1), SPECIAL EFFECTS (5), STUNTS (14), SOUND (4), COSTUMES (1), MAKEUP (2), SET (2)) is the single biggest contributor to the "admin panel" feeling. This is inventory management UI — it belongs in a production management view, not in the writer's primary sidebar.

  • The breakdown accordion as the default sidebar content. Category names with counts in parentheses is a database interface pattern. During writing, this data is noise. It should be available — but as a dedicated breakdown view, not as the default writing companion.
  • The top action bar. "Scene Breaks · Download Script Synopsis · Share Project · Script AI Dude · Export" — this is a feature showcase toolbar. Every button competes with every other button. Most of these are occasional actions (export, download, share) that don't deserve permanent toolbar presence.
  • "ADD NEW CATEGORY" at the bottom of the sidebar. This is a CMS pattern. Writers don't add categories; production staff might, and that's a platform concern.

What Feels Too Technical

  • The "3 4/8" page count format. While this is industry-standard (pages in eighths), its presentation currently feels like a raw data readout. It should be formatted more gracefully — perhaps with a subtle label: "Page 3⅜ of 8" or simply positioned in the status bar with clear typography rather than floating in the top-right corner.
  • The tag popup ("CAST" badge). The inline tag that appears when hovering over a character name is informational but looks like a developer tooltip. Tags should feel integrated — either as subtle inline marks (already partially achieved with the color underlines) or as inspector content when the writer explicitly requests detail.

What Feels Distracting

  • Color saturation of inline tags. The combination of bold purple character names, green locations, amber props, pink SFX markers, and yellow highlights creates visual competition with the screenplay text itself. The tags are louder than the writing. Direction: tags should use lower-opacity backgrounds and thinner underlines. The text content remains primary; tags are annotation, not decoration.
  • Multiple navigation FABs in the bottom-right. The floating action buttons (navigate forward/back, grid icon, chat icon with notification badge) add visual clutter to an area that should be quiet. These should integrate into the shell chrome or be accessible via keyboard/command palette.

What Screenplay Writing Should Emotionally Prioritize

  1. Flow. The act of writing — entering text, formatting it correctly, moving between blocks — must be the fastest, most frictionless experience in the product.
  2. Structure. Awareness of where you are in the script (which scene, which act, how many pages) should be ambient, not effortful.
  3. Story. Characters, locations, the emotional arc — the breakdown data exists to serve story understanding, not inventory counting.
  4. Production readiness. The awareness that this script will eventually be shot. Formatting matters. Scene numbers matter. But this awareness should be quiet confidence, not prominent UI.
Direction

The screenplay page must remain the center of gravity. Tags are whispered annotations — visible when you look for them, never louder than the prose. The sidebar serves scene navigation first, breakdown second. Production data is a deeper layer — accessed intentionally, never imposed.

Section 5

Inspector & Intelligence Philosophy

The Inspector as Contextual Companion

The inspector panel is the editor's primary intelligence surface. It is not a permanent sidebar. It is a responsive companion that shows information relevant to the writer's current focus. Its default state is collapsed — the writing surface uses the full width. When opened, it provides depth without disruption.

The inspector's contents are always driven by context:

Writer's Focus Inspector Shows
Cursor in scene Scene notes, scene-level metadata (location, time, estimated duration), scene-level AI observations
Character name selected Character card — occurrences, relationships, wardrobe notes, first/last appearance
Tagged prop/location Prop/location detail — description, occurrences, continuity notes
Nothing specific Script-level overview — structure summary, character list, recent notes, revision state
AI conversation active AI panel — variations, critique, or breakdown results for the selected passage

Intelligence Positioning

The doctrine is clear: AI assists, never authors. The UX implication: AI never produces finished prose in-line. It never auto-completes. It never suggests while you type. Instead, AI is invoked — the writer asks, and the AI responds in a contained surface (the inspector panel or a focused overlay).

Intelligence Behavior Spectrum
Correct Writer invokes AI.
AI offers options.
Writer chooses or dismisses.
Acceptable Editor surfaces an observation.
"Scene 4 has no location details."
Writer decides to act or ignore.
Forbidden AI writes text into the document.
Autocomplete appears while typing.
Suggestions overlay the page.

How AI Should Feel

  • Quiet. No animated AI avatars. No pulsing "thinking" indicators that demand attention. When AI is processing, a minimal progress indicator in the inspector is sufficient — never a modal overlay.
  • Contextual. AI responses appear in the inspector, anchored to the passage or element the writer was focused on. They don't float. They don't pop up.
  • Respectful. AI offers variations (multiple alternatives, never one "answer"), critique (observations and questions, never rewrites), breakdown (structural understanding offered, never imposed), and ask (conversational lookup of the writer's own material).
  • Optional. Every AI surface can be dismissed. No AI feature is required to use the editor. The editor functions fully without any AI engagement.
  • Non-invasive. AI never modifies the document without explicit writer action. No "accept suggestion" flows that modify text automatically. The writer copies, adapts, or types — always manually.

Intelligence Visual Language

AI-originated content should have a distinct but quiet visual identity. The existing design system uses an amber accent for AI-related surfaces (AI progress indicators, thinking states). This is correct — amber communicates "supplementary intelligence" without the connotations of blue (corporate AI) or green (success/completion). AI content in the inspector should use:

  • A subtle amber left-border on AI-generated content blocks
  • Slightly reduced text opacity compared to writer-authored content
  • Clear provenance labels: "AI Variation" / "AI Observation" — never unlabeled
  • The existing rw-ai-thinking pattern for processing states — subtle pulse dots, not dramatic animations

Inspector Behavior Rules

  • Never auto-open. The inspector opens only when the writer explicitly requests it (keyboard shortcut, rail icon, right-click context action).
  • Context-switch smoothly. When the writer moves their cursor from one scene to another, the inspector's content updates — but without animation or transition that draws attention. The content simply changes.
  • Remember state per session. If the writer opened the inspector to the notes panel, it stays on notes until they change it. Per-session memory, not per-cursor-position switching between panels.
  • Respect the page's gravity. The inspector never exceeds 280px width. It uses compact typography. It is dense but clear — the style of a reference margin, not a feature panel.

Section 6

Filmustageation Visual Direction

Color Philosophy

The existing Rwanga palette is strong and should be preserved. The dark theme (#0F0F12 base) is the correct foundation for a writing environment — it reduces eye strain during long sessions and makes the page itself feel luminous by contrast. The warm amber accent (#D4A574) correctly avoids the cold blue accent trap that makes tools feel corporate.

Direction adjustments:

  • The page surface should have a subtle warmth — not pure white or pure black, but a warm dark (#1A1A1A range) that suggests paper under low light. The page is not a browser viewport; it is a manuscript surface.
  • Tag colors should be reduced in saturation by 30–40% when inline in the document. Full-saturation tags compete with the writing. Tags in the sidebar or inspector can use full saturation because they're separated from the prose.
  • The section color system (Script orange, Breakdown blue, Visualize teal, Plan purple, Shoot pink) should remain — but as environment tinting, not element coloring. The status bar tints. The rail indicator tints. A panel header might tint. But the writing surface never tints.

Contrast Philosophy

Three contrast zones:

  • The page — highest contrast. Text is bright against dark. This is where the eye lives.
  • Chrome — medium contrast. Sidebar, inspector, status bar use muted text against slightly lighter surfaces. Readable but not attention-grabbing.
  • Ambient elements — lowest contrast. Rail icons, inactive tab labels, background metadata — these exist at the edge of perception. Visible when sought, invisible when not.

Typography Feeling

Two type worlds:

  • The writing surface: Courier Prime (or equivalent Courier). 12pt. Fixed-pitch. Double-spaced. This is non-negotiable for screenplay; it's the industry standard. The monospace grid is the page's skeleton — it communicates "this is a professional screenplay, not a blog post."
  • The chrome: Inter (already in the design system) or the system font stack. Variable-width. Compact sizes (10–13px). The chrome type should feel like instrument labeling — precise, small, subordinate to the content it describes.

The type contrast between these two worlds is itself a design feature. The jarring difference between Courier on the page and Inter in the chrome reinforces the boundary between content and environment.

Motion Philosophy

Motion in the editor should be functional, not decorative. The existing design system specifies 0.12s ease transitions — this is correct. Motion serves three purposes:

  • State confirmation. A sidebar collapses with a quick slide. A panel opens with a brief fade. These confirm "something happened" without demanding attention.
  • Spatial continuity. When the inspector opens, the writing surface narrows smoothly — not instantly. This maintains spatial awareness ("the page is still here, just sharing space").
  • Focus transitions. When entering deep-writing mode, chrome fades out over ~200ms. This is slow enough to feel intentional, fast enough to not feel theatrical.

No motion should exist for its own sake. No loading animations beyond minimal spinners. No micro-interactions on buttons (the current opacity 0.88 hover is sufficient). No parallax, no elastic bounce, no spring physics.

Spacing Philosophy

The editor uses architectural spacing — spacing that creates rooms, not margins. The writing surface has generous padding (1 inch top, 0.5 inch sides in the current spec) that creates a "page" within the viewport. Chrome surfaces use tighter spacing (8–16px internal padding) because they are utility spaces, not creative spaces. The contrast in spacing density between page and chrome reinforces their different roles.

Paper vs. Chrome Relationship

Paper / Chrome Attention Balance
Rail
Sidebar Low attention Glanceable
The Page Maximum attention Sacred territory Highest contrast
Inspector On-demand Contextual depth

The page is always the brightest, most spacious, most typographically generous surface. Chrome surrounds it at lower contrast, tighter density, smaller type. This relationship should be perceptible even in screenshots — a stranger glancing at the editor should immediately understand: "the thing in the middle is what matters."

"Studio" Feeling

The overall atmosphere should evoke a professional workspace at night — controlled lighting, clean surfaces, purposeful arrangement. Not a tech startup (bright colors, playful shapes). Not a film set (dramatic, chaotic). Not a library (dusty, analog). A studio: where skilled people make things under good conditions.

The design system's near-flat radius (2px) contributes to this — rounded corners signal friendliness; squared corners signal precision. The editor is precise. The total absence of box-shadows (already enforced in the design system) contributes further — shadows suggest floating; the editor's surfaces are placed, architecturally, not hovering.

The Spectrum of Attention

Writing is not one activity. It moves between states — sometimes within minutes. The editor must accommodate each state without requiring the writer to explicitly "switch modes." The transitions should be organic: the writer's behavior signals which state they're in, and the editor responds.

Deep Writing

The most important mode. When the writer is composing — typing new text, editing dialogue, crafting action lines — the editor should shed everything that isn't the page. This doesn't require a manual "focus mode" button (though one should exist as a shortcut). It means: when the sidebar and inspector are closed, the writing surface expands to fill the available space with generous margins. The status bar dims to near-invisibility. The title bar shows only the script name. The writer's world is: their words, the cursor, the page.

Review

When the writer shifts from creating to evaluating — reading through a sequence, checking continuity, adding notes — the chrome should feel welcoming. The sidebar shows scene navigation for quick jumping. The inspector opens to notes or revision history. Markup and revision colors appear on the page. The environment says: "you're reading now, here's context to help."

Breakdown

The production-analysis mode. This is where the breakdown accordion currently lives — and where it should live. During breakdown work, the sidebar switches to the category view (characters, props, locations, etc.), tag highlights on the page increase in visibility, and the inspector shows detailed tag information. This is the mode where the "admin panel" feeling is appropriate — because the writer has shifted to production thinking. The key: this mode is entered intentionally, not imposed by default.

AI Assistance

When the writer invokes AI (selecting a passage and requesting variations, critique, or breakdown), the inspector opens to the AI panel. The selected passage remains visible on the page with a subtle highlight. The writer's attention splits between the page (source) and the inspector (AI response). This is a dialogue — the writer asks, the AI responds, the writer acts or dismisses. The AI panel is never the primary surface; it's always secondary to the page.

When Chrome Fades

Chrome fades when the writer's behavior signals deep creation: sustained typing without navigation, no sidebar or inspector interactions for more than a few minutes, cursor staying within a single scene. The editor doesn't literally hide chrome on a timer — but it should never add chrome during writing flow. No notifications, no "you've been writing for an hour" badges, no proactive suggestions. Silence is the editor's deepest feature.

When Information Expands

Information expands when the writer explicitly requests it: clicking a rail icon to open the sidebar, pressing a keyboard shortcut to open the inspector, right-clicking a tagged element to see its details, invoking the command palette. Every information expansion is writer-initiated. The editor never decides "you probably want to see this now."

Section 8

Future Collaboration Direction

Doctrine Anchor

Collaboration features live at the platform layer, not in CORE. CORE provides revision anchoring as a primitive. The platform decides who sees what. — §4.2, §7.5

Emotional Direction for Collaboration

Collaboration in a writing environment is fundamentally different from collaboration in a spreadsheet or design tool. Writing is intimate. A screenplay is a creative act, not a group deliverable. Collaboration features must respect this intimacy.

Comments

Comments should feel like margin notes in a printed script — small, anchored to a specific passage, visible but not intrusive. Not chat bubbles. Not threaded conversations with avatars and timestamps dominating the view. A comment is a pencil mark in the margin: you notice it, you read it, you respond or dismiss. The writing remains the page's primary content. Comments are annotations — secondary, subordinate, peripheral.

Reviews

A review is a structured reading pass. The reviewer reads the script with the ability to annotate — approving, questioning, or requesting changes on specific elements. The review UI should feel like a reading experience with annotation capability, not a task-management interface. No "Approve" / "Request Changes" buttons that look like Jira tickets. Instead: margin marks, inline notes, a summary that the writer can process at their own pace.

Collaborative Notes

Notes are the most natural collaboration surface for screenwriting. Directors leave notes. Producers leave notes. Actors leave notes. The editor should treat notes as layered voices — each contributor's notes are attributed and distinguishable, but all are subordinate to the script itself. Notes never modify the document; they annotate it.

Production Feedback

Production feedback (budget concerns, scheduling conflicts, location availability) is platform-level context that may need to surface in the editor. When it does, it should appear as structured metadata in the inspector — not as inline comments competing with creative notes. A production concern about a location is a different category of information than a director's note about a character's motivation. The UI should distinguish these categories through visual language (perhaps using the section color system: production feedback uses the Plan purple, creative notes use the Script orange).

Revision Tracking

Screenplay revision tracking has industry conventions: revision colors (white, blue, pink, yellow, green, goldenrod, buff, salmon, cherry, tan, second blue…). The editor should respect this convention while making it feel integrated rather than overlaid. Revision colors should appear as subtle page tinting or margin indicators — not as tracked-changes redlines competing with the readable text. The writer should be able to view any revision draft cleanly, without markup — and then overlay the revision markers when comparing.

The Principle

All collaboration features share one principle: the script is sacred ground. Collaboration happens around the script — in margins, in panels, in overlays. Never inside the script text. Other people's voices enrich the writer's awareness; they never invade the writing surface.

Closing Assessment

Strengths, Risks, and Future Work

01

Strongest Current UX Strengths

  • The writing surface is fundamentally sound. Proper Courier typography, correct screenplay block formatting (action/character/dialogue/parenthetical with appropriate indentation), scene-header structure. The page feels like a screenplay page. This is the foundation — it must not be disrupted.
  • Inline breakdown tags are the right instinct. Surfacing production data (characters, props, locations) as color-coded highlights within the writing itself — rather than in separate panels — is the correct approach. It keeps the writer in context. The execution needs refinement (lower saturation, less visual competition with prose), but the concept is strong.
  • The design system has excellent bones. The dark palette with warm amber accent, near-flat 2px radius, no-shadow philosophy, section color system, Inter/Noto Sans Arabic type stack — all of this is mature, intentional, and well-suited to a professional creative tool. The evolution should build on this vocabulary, not replace it.
  • The scene navigator exists and works. The left sidebar's scene list — numbered scenes with slug-line previews, collapsible by act/sequence — is useful and follows correct screenplay workflow. It needs to be elevated to the sidebar's primary role (currently it shares space with the breakdown accordion, which dilutes both).
  • The dark-theme-first approach is correct for a writing environment. Extended sessions benefit from reduced eye strain. The warm dark base avoids the cold, institutional feel of pure-black dark modes.
02

Biggest UX Risks

  • Platform chrome contaminating editor identity. The top navigation bar (Script / Scheduling / Call Sheets / Budgeting / Storyboards) is platform-level navigation that currently lives inside the editor viewport. As the platform grows, this contamination will worsen — more tabs, more features, more chrome that has nothing to do with writing. The editor must have its own viewport boundary that the platform wraps, never penetrates.
  • The "admin panel" sidebar. The breakdown accordion (CAST (5), PROPS (8), etc.) as the default sidebar creates an inventory-management first impression. If this remains the default view, every new user will perceive the editor as a production management tool rather than a writing environment. The fix is simple (default to scene navigator; move breakdown to a secondary view) but must happen before the pattern calcifies further.
  • Tag visual competition with prose. The current tag highlighting (bold + color + background) makes tagged words visually louder than surrounding prose. In a script with many tagged elements, the page becomes a coloring book rather than a screenplay. Tags must be subordinate to prose — always. The writer needs to be able to control tag visibility (off / subtle / visible) as a personal preference.
  • Feature-toolbar creep. The current action bar (5 buttons plus navigation FABs) will grow as features are added. Without a strong containment strategy (command palette as primary access, toolbar limited to 2–3 context-sensitive actions), the toolbar will become an unmanageable strip of competing buttons — the canonical SaaS admin-panel pattern.
  • AI surface competing with writing. If the "Script AI Dude" button evolves into a persistent sidebar, chatbot panel, or inline suggestion system, it will directly violate the doctrine's "AI assists, never authors" principle. The AI access pattern must be established now — invoked, contained, dismissible — before product pressure pushes toward more aggressive AI presence.

Areas Needing Future Engineering Cooperation

  • Inspector panel reactivity. The contextual inspector (content changes based on cursor position / selection) requires tight integration with the ProseMirror editor state. Engineering must define how cursor-position events propagate to the inspector panel without performance cost.
  • Plugin slot injection system. The sidebar, inspector, status bar, and toolbar all accept plugin-contributed content. Engineering must build the injection-point API before the second plugin exists, so the contract is tested by both the screenplay plugin and a neutral test plugin.
  • Focus-mode state management. Deep-writing focus (minimal chrome) vs. breakdown mode (full chrome) vs. review mode (markup visible) requires a state machine that coordinates sidebar visibility, inspector state, tag visibility, and toolbar contents. This should be CORE infrastructure, not per-plugin logic.
  • Tag visibility controls. The ability to toggle inline tag visibility (off / subtle / visible) per tag category and globally requires coordination between the ProseMirror schema decoration system and a user-preference layer. Engineering should design this as a CORE capability that plugins configure.
  • Platform / editor viewport separation. The editor must render inside a bounded viewport that the platform chrome (navigation, project management, account UI) wraps. This is likely an iframe or a well-defined DOM boundary. Engineering must define this boundary before the platform grows further.
04

Areas Requiring Future Visual Exploration

  • Tag visual design at reduced saturation. The current full-saturation tags need to be redesigned at 60–70% saturation with thinner underlines and optional backgrounds. This requires visual exploration — the tags must remain distinguishable from each other while being subordinate to the prose. Color accessibility across the dark and light themes adds complexity.
  • Inspector panel content design. The contextual inspector has many potential content types (scene notes, character cards, AI variations, revision history, breakdown detail). Each needs a distinct but harmonious visual treatment within the inspector's 280px constraint. This is a significant design surface that needs its own exploration phase.
  • Focus-mode transitions. How exactly does chrome fade during deep writing? How does the status bar dim? How does the sidebar collapse feel? These transitions are small but define the editor's personality. They need motion-design exploration — not as spec, but as feel.
  • New-document type chooser. When multiple plugins coexist, the New Document flow must offer doc-type selection. This is a first-impression surface — it communicates "this editor supports multiple writing forms" without overwhelming. Needs careful visual exploration to avoid both under-designed (plain list) and over-designed (wizard/onboarding) extremes.
  • Screenplay-specific page chrome. Scene headers, act breaks, page breaks, MORE/CONT'D markers — these are screenplay-plugin-contributed page elements that live on the writing surface. They need visual design that's distinct from body text but doesn't feel like UI widgets on the page. The current scene-header styling (yellow left-border, tinted background) is a starting point but needs refinement.
  • Collaborative annotation visual language. Comments, review marks, production notes, revision indicators — each needs a visual identity that's distinguishable from the others and from the writer's own content. This is a future exploration that should happen before the platform's collaboration features are implemented.
