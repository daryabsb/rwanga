Semantic Entity Layer — Surface Direction

Alias UX Direction

The writer's path from a mention to an identity. One thing only: how a writer creates and understands an alias — not Inspector, not Timeline, not AI, not Character Profiles, not a merge tool. Those surfaces read identity. This designs the one place a writer writes it.

status: direction-only grounded-in: tag-system.js · post-f1a-review · sel-doctrine-lock schema: untouched date: 2026-06-05

The model, in the writer's words

One entity. Many names.

The architecture is already three links long. A mention is a tag span in the manuscript — the actual words on the page. An entity is a registry record. The resolver follows the mention's id to the entity, and is now alias-aware.

What is missing is the writer's hand on the middle link. Today tagSelection('character', 'The Teacher') mints a new entity called "The Teacher." There is no way to say "The Teacher" means Nali.

Doctrine — Alias ≠ Merge

We never create a duplicate entity and then fold it. We tag the mention straight onto the existing entity and remember the surface form. Nali · The Teacher · The Old Man · The Poet — one identity, four surface forms. There is nothing to undo, reconcile, or de-duplicate.

Boundary — pronouns are out of scope

He · She · They · Him · Her are not part of this work — they are future context-resolution. This package designs distinctive aliases only: The Teacher, The Butcher, The Old Man, The Driver, The Doctor.

01 · Recommended UX

"Tag As," deepened — not a second verb

The writer already knows one move: select text → right-click → Tag "…" as → [Type]. We do not add an Alias Of verb. We deepen the verb that exists.

Choosing the type already narrows the candidates — a writer aliasing "The Teacher" knows they mean a character. So the type submenu is exactly where the candidate list belongs. When the type already has entities and the selected text isn't an exact match of one of them, the type opens an Entity Picker as its submenu.

02 · How the writer understands an alias

Three escalating surfaces — page, sidebar, inspector

Color in this system is per type, not per entity (Rga.Color.getTagColor(type) — every character is #4FC1FF). Color can never tell two characters apart. Identity has to be carried elsewhere, on three surfaces the writer escalates through only if they go looking.

03 · The smallest UI that teaches identity

The picker is the teaching surface

At the exact moment the writer tags "The Teacher," seeing New Character sitting above Nali · Baban · Maryam poses the question — new person, or someone I've met? — in situ, with no screen, no manager, no admin. The toast then narrates the answer in one sentence. Discovery escalates only if the writer goes looking: page marker → sidebar line → Inspector field. Nothing is pushed; everything is glanceable.

What we refuse to build

No alias manager. No merge dialog. No duplicate-resolution queue. No entity database table. No admin screen. Filmustageation is writer-first; entity internals stay internal.

04 · Rejected directions

What we considered and refused

#OptionWhy rejected
R1A separate Alias Of → verb beside Tag AsForces the writer to decide new vs. alias before seeing candidates; doubles the menu; leaks the entity/alias internal distinction into the writer's vocabulary.
R2A dedicated "Manage Aliases" dialog / screenA database manager. Violates writer-first doctrine; exposes entity internals; turns identity into administration.
R3Auto-alias by fuzzy / AI name matchingSilent identity decisions are dangerous — identity is always the writer's call. (Pronoun resolution is explicitly future work.)
R4Drag one tag onto another to aliasInvisible, undiscoverable affordance — and it reads as a merge, exactly the mental model doctrine forbids.
R5An "☑ this is an alias of…" checkbox in the create dialogAdds modal friction to the most frequent action (tagging) to serve the rarer one (aliasing). Inverts the cost.

05 · Future compatibility

Write identity once; every other surface only reads it

The whole design rests on one discipline: the alias action writes identity to exactly one place — entity.aliases plus the mention's tagId — and every other surface only ever reads identity by entity id. Because no consumer reads surface strings, none of them needs the alias workflow to change when they ship.

Future surfaceWhat it inherits for free
Character ProfilesThe profile is the Inspector entity view grown up. aliases is already populated; "Also known as" is already rendered.
InspectorAlready entity-scoped. Renders aliases; no new contract.
TimelineScene presence is computed per entity id. "The Teacher" in scene 12 already counts as Nali present — no alias-specific code.
AIReads { id, canonical, aliases[] } from the resolver, never raw strings. Four phrases are one character with zero prompt engineering.
MCPExposes the same entity record. Identity travels across the boundary intact.
The load-bearing restraint

This design does not touch schema.marks.tag — the surface the F1A review flags as dangerous and load-bearing (every saved .rga depends on its shape). Aliases are an additive field on the registry record, written through the existing addEntity path. No new mark, no .rga migration for new files.

06 · Implementation guidance for engineering

Eight touch-points in tag-system.js — direction, not code

#SurfaceChange
1_createEntityAdd aliases: []. Thread through serializeRegistry / loadRegistry. Default [] on legacy load.
2_buildTagTypeSubmenuEach type item becomes a submenu (the Entity Picker) when entities of that type exist and the text isn't an exact match. First item New {Type}: "{text}"; then existing entities (dot + name + count); filter input past ~7 rows.
3tagSelection(type, text, entityId?)Add optional entityId. When present and text ≠ entity.name (case-insensitive), entity.aliases.push(text) (dedupe). When absent, today's find-or-create.
4ResolverAlready keyed by tagId; make alias lookups read entity.aliases so name→entity resolution finds them. No schema change.
5Inline render.tag-highlight[data-alias="true"] → dotted bottom border; tooltip "{TypeLabel}: {entity.name} · alias".
6updateManagerPanelRender the also: line when entity.aliases.length.
7_showInInspectorAdd the Also known as field — removable chips, per-alias counts.
8Do notAdd a mark, a tagRegistry key, or a migration. aliases rides on the existing per-entity record.

07 · Risks & open questions

RiskMitigation
Submenu depth — a third right-click levelFilter past ~7 entities; full keyboard nav; the picker only opens when a decision is actually needed (fast path otherwise).
Accidental alias (wrong entity picked)Undo restores prior state; the toast names the entity so the error is caught immediately; the Inspector chip is one click to remove.
Per-type color collisionBy design — color means type. Identity lives in the marker, tooltip, and sidebar nesting. Do not introduce per-entity color; it breaks the type-color contract the editor relies on.
Pre-existing duplicate entitiesSee §08 — a bounded, opt-in reconcile affordance, explicitly not the alias-creation flow and not a routine verb.
Removing an alias — what about its mentions?S2: removing an alias detaches its mentions (they revert to plain text) with undo. Never silently re-points or deletes the entity.
Pronoun boundaryHe / She / They are out of scope — future context-resolution. This package designs distinctive aliases only.

08 · Migration path toward Character Profiles

Two things must stay true

A — Reconcile, not merge (bounded, opt-in)

Files written before this shipped may contain two entities that are really one person (a "The Teacher" character minted because there was no alias path). This is the only place anything merge-shaped is permitted, and it is framed as repair, not a feature: from an entity's Inspector, "This is actually… → Nali" re-points that entity's mentions onto Nali and folds its name into Nali.aliases. It is not in the right-click tagging flow, not a routine writer verb, and not surfaced as "merge." It cleans up the pre-alias past, once, and then recedes.

B — The Inspector entity view is the Profile seed

"Also known as" (§02) is the first Profile field. When Profiles ship, they are the same entity record with more fields (description, relationships, arc) — the alias workflow underneath is unchanged, because it only ever wrote identity.
